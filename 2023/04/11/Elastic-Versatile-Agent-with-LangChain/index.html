遥想当年 AlphaGo 打败人类顶尖围棋选手¶

在 AlphaGo 于 2016 年击败顶尖职业棋手之后，许多围棋职业选手的『出路』主要体现在以下几个方面：

  1. 部分棋手选择退役或转型
  • 李世石在 2016 年 3 月与 AlphaGo 的五番棋比赛中以 1:4 失利后，于 2019 年 11 月宣布退役，理由是『无法再在围棋上达到顶峰』。
  • 柯洁在 2017 年 5 月 0:3 不敌 AlphaGo 后，转而深入研究 AI 对局，从中汲取新思路，并在此后对人类对弈大幅获益，甚至一度取得 22 连胜。
  1. 专业棋手越来越多地投入教学与解说
  • 职业棋手通常分为『竞赛派』和『教学派』两类。竞赛派以参加各类赛事实物奖励为生；而教学派则通过解说电视、撰写围棋书籍、开设围棋学校、线上授课和直播等多元化方式获得收入。
  • AlphaGo 的出现反而刺激了教学需求：大众希望通过 AI 训练提升棋力，教学派棋手借此推出了 AI 驱动的训练课程和『一对一 AI 辅导』服务。
  1. 利用 AI 工具提升自身和学员水平
  • 围棋职业选手开始将 AI 对弈引擎（如 Leela Zero、KataGo 等）纳入日常训练，通过解析 AI 托底的『最佳着法』和『定式创新』，大幅加快学习进度。
  • 一项研究表明，职业棋手在与 AI 共同复盘（『双人围棋』）后，其棋力平均提升了数个段位，AI 不再是对手，而是成了『最强训练伙伴』。
  1. 跨界进入 AI 与科技领域
  • 有的顶尖棋手受邀参与人工智能研究或 AI 应用产品开发，比如担任 AI 围棋项目的咨询顾问或算法评估专家。
  • 也有棋手转向内容创作（YouTube 频道、知识付费平台）以及赛事与俱乐部运营管理，利用自身影响力为围棋生态持续赋能。

Google DeepMind 的 AlphaEvolve

让我们从 Google DeepMind 开始，这家公司的「Alpha」系列开发长期以来一直在革新 AI 领域。你是否还记得传奇性的 AlphaGo？它是第一个在复杂棋类游戏围棋中击败世界冠军的 AI 系统。随后是 AlphaZero，这是一种通用型游戏 AI，不依赖人类数据就能掌握国际象棋、将棋和围棋。它通过自我对弈学习，展现出蒙特卡罗树搜索（MCTS）算法在决策下一步棋时的强大能力。

其中最杰出的突破之一是 AlphaFold，它解决了生物学中最大的难题之一：如何仅凭氨基酸序列预测蛋白质的三维结构。得益于机器学习、进化生物学与结构建模的结合，它彻底改变了我们理解蛋白质的方式，节省了数年的实验室工作时间。AlphaFold 预测了超过两亿个蛋白质的结构，加速了药物研发与医学研究。最终，AlphaFold 的两位主要开发者 Demis Hassabis 和 John Jumper 因此获得了诺贝尔化学奖。

Google DeepMind 在问题求解工具方面的成果也同样值得关注。AlphaGeometry 擅长解决复杂几何问题，其表现可与数学奥林匹克中的顶尖人类选手媲美。AlphaProof 则结合语言模型与强化学习技术，将自然语言问题转化为形式化证明，其水平相当于国际数学奥林匹克的银牌选手。

可见，Google DeepMind 的开发成果不仅限于人工智能，还服务于我们世界中的关键实际领域，并构建出最强大的自我学习算法。而现在，轮到全新的 AlphaEvolve 登场了。

AlphaEvolve 是 DeepMind 提出的一种用于自动发现算法或数学程序的 AI 系统。它结合了大型语言模型（如 Gemini）、进化算法和强化学习（AlphaZero 风格），能在无需人类指导的情况下自动『进化』出解决复杂数学或算法问题的程序，例如整数因式分解、排序等。

AlphaEvolve 本质上是一个先进的 AI 工具，能够帮助大语言模型（LLM）在解决复杂问题（如改进计算机系统或应对科学挑战）方面变得更强。它是一个受到「进化」概念启发的编程代理系统，其工作原理类似自然选择：不断尝试代码变化、获取反馈，并持续优化。这个工具实现了科学发现过程中一部分的自动化，包括头脑风暴、实验尝试、从失败中学习并精炼结果。这个过程往往需要数年时间，但通过让 AI 来完成大量工作，AlphaEvolve 大大加快了这一进程。

在过去一周里，我反复阅读了《AlphaEvolve》这篇论文，每次阅读都能发现新的、令人着迷的细节。毫不夸张地说，我认为 AlphaEvolve 可能是去年最重要的 AI 突破之一。通往通用人工智能（AGI）的道路可能有多种形式，其中一种可能就是 AI 自主发现新的 AI，也就是发现新的算法——而这正是 AlphaEvolve 的核心所在。那么，它是如何做到的？让我们深入探讨。

进化算法与大型语言模型（LLMs）的融合，正在开启科学与算法发现的前所未有的新篇章。在 DeepMind 最近发布的白皮书《AlphaEvolve：用于科学与算法发现的编码智能体》中，他们提出了一种全新的智能体架构，它在一个进化框架中协调 LLM 的运作，实现了对算法的自主生成与优化。通过引入测试时的反馈循环（test-time feedback loops）并利用前沿的大型语言模型，AlphaEvolve 不仅产生了在数学上具有重要意义的结果，还为 Google 的生产基础设施带来了实际改进。

参考围棋界的经验，以及当前 AI 发展的趋势，可以总结出程序员在被 AI Agent 打败之后可能的出路有以下几种方向：

  1. 转型为 AI 教练、指挥或系统集成专家
  • AI Agent 本身虽然能编程，但它们仍需要人类规划大目标、设定多阶段任务、检查整体逻辑。
  • 顶级程序员可以转向设计复杂系统架构，担任『AI 指挥官』，不是亲自写代码，而是让 AI 写并做高级审核与集成。
  1. 深化与 AI 的协作，成为超级个体
  • 像过去用 IDE 一样自然地用 AI Agent，单人能完成原本需要几十人才能完成的项目。
  • 精通 Prompt Engineering、AI Agent Workflow 设计，成为『AI 助力型开发者』。
  1. 转向 AI 不擅长的领域
  • AI 在常规编程任务上非常强，但在人类价值判断、伦理设计、跨文化沟通、用户体验设计等领域依然有短板。
  • 程序员可以转型为偏向产品经理、伦理顾问、用户体验工程师等角色。
  1. 投身 AI 工程本身
  • 参与 AI Agent 的训练、优化、评价体系建设，比如设计更好的训练数据集、设定评估标准、控制 hallucination（幻觉现象）。
  • 成为『AI 编程教练』：训练和提升 AI 编程系统，就像今天训练一只 AlphaZero。
  1. 教育与传播
  • 培养下一代人类程序员，或者教授大众如何使用、审查、定制 AI 程序员。
  • 围绕 AI 编程工具建立新的教育生态，比如 Prompt 设计、Agent 协作课程。
  1. 跨界与创业
  • 结合 AI 编程能力，跨界进入新的行业（医疗、金融、能源、制造业），用『程序员思维 + AI 工具』开创新商业模式。
  • AI 本身会拉低程序开发门槛，但真正懂得行业痛点的人依然稀缺。

Model Context Protocol¶

Microsoft TaskMatrix

1. LLM 本地部署与推理¶

  • 利用 Apple Silicon 原生加速，你可以部署量子化的轻量级 LLM 模型，如：
    • llama.cpp（支持 M 系列 GPU 和 Neural Engine）
    • mlc-llm（专为 Apple Silicon 优化）
  • 利用 Metal + Neural Engine，可以进行高效推理，适合开发和测试。

2. 机器学习与神经网络开发¶

  • 使用 Apple 提供的 Core ML 框架，让模型部署和运行充分利用 Neural Engine。
  • 用 Python + PyTorch 或 TensorFlow + MPS（Metal Performance Shader） 后端训练和微调中小型模型。
  • 推荐关注 ML Compute —— Apple 对机器学习任务的加速支持。

MCP Playwright 客户端框架比较¶

以下表格总结了 Semantic Kernel、LangChain、AutoGen、mcp-agent 和 fast-agent 这五个框架在关键维度下的特点对比：

Semantic Kernel 原生支持MCP（Python 1.28+、.NET 均可）。示例代码通过 npx @playwright/mcp@latest 启动 Playwright MCP 服务，并自动将其函数映射为 Kernel 插件。 提供 C#（NuGet）和 Python（pip）SDK，需要相应运行时。.NET 版集成 Azure/OpenAI；Python 版与常规 Python 环境兼容。 支持 Azure OpenAI、OpenAI 等云端模型，并提供本地模型连接器（如 Ollama）。 以单 Agent 或函数调用为主，缺乏内置的多代理编排。用户可通过函数编排实现链式调用。 支持插件模式，可将外部工具/函数打包为 KernelFunctions，生态扩展主要依赖社区实现的 Connector（如 Ollama Connector）。 微软官方支持，文档齐全且权威。社区规模中等（较 LangChain 小），但有逐渐扩大的用户群。 C# 版执行效率高，Python 版性能与常见 Python 框架相似。总体资源开销适中。 适合企业级部署和成熟场景，尤其是在 .NET/Azure 生态中构建稳健的 AI 应用。对于需要调用 Playwright 等工具的自动化应用非常便利。
LangChain 非原生 MCP 客户端，但可通过官方 langchain-mcp-adapters 库访问 MCP 服务。可将任意 MCP 服务器定义的工具集成进 LangChain Agent。 Python 框架（pip 安装），依赖较多（涉及各类 LLM 库和工具），但安装和环境配置流程成熟。 原生支持 OpenAI、Anthropic 等主流云服务，还可利用 Hugging Face Pipeline 本地运行模型。提供多种 LLM 包装器（OpenAI、Ollama、HuggingFace 等）。 支持复杂的多代理架构（LangGraph 多Agent 系统），提供网络拓扑、监督者、层级等多种协作模式。易于搭建串行/并行链式工作流。 框架组件模块化，拥有 600+ 集成插件，几乎可对接所有常见服务和数据库。用户可自定义 Chain、Tool、Memory 等，多层次的扩展性极佳。 社区规模最大（3000+ 贡献者，600+ 集成），文档丰富（官方教程、示例、博客等）。支持 Python/JavaScript 两种语言，社区资源和生态最为完善。 纯 Python 实现，性能依赖具体用法和并发控制。一般情况下可通过 LangSmith 等工具监控与优化。并非针对轻量化极限优化，但在生产环境已被多家公司验证可用。 适合快速原型开发和生产部署，尤其是需要灵活构建复杂链式推理或多模型组合场景。大社区支持易于入门，适合研究和业务验证。
AutoGen 原生未直接支持 MCP，但社区提供了扩展（如 autogen-ext-mcp）。可通过 Agent 中的接口调用外部工具（需手动接入 MCP 客户端）。 Python 框架（pip 安装 autogen-core），要求 Python 3.10+。另需安装 OpenAI 扩展（autogen-ext[openai]）以使用云模型。 支持 OpenAI/Azure OpenAI（云服务），并支持本地推理（如 LiteLLM、Ollama）的接入。可灵活切换模型后端。 内置多 Agent 设计：支持“GroupChat”等模式，可在同一会话中组合多个智能体协同工作。还提供助手 API，实现多技能协同（代码执行、RAG检索、函数调用等）。 AutoGen 提供多种组件（Agent、技能、工作流）。可定制工具（如 CodeInterpreter）、状态管理和技能库，具备一定的插件式架构。 由微软维护，文档涵盖教程和示例，官网和 GitHub 文档可用。社区相对较小，主要以学术和内部使用者为主，但有官方示例和案例支撑。 Python 实现，基于代理通信和上下文缓存，资源开销依赖于模型大小和并行度。具有良好的扩展能力，但为了多 Agent 通讯可能存在额外开销。 适合需要多智能体协作或人机混合交互的场景（例如可视化工作流、教育或研究用途）。借助 AutoGen Studio 可快速构建复杂 Agent 系统。
mcp-agent 原生支持 MCP 协议，是为此设计的轻量级框架。自动管理多个 MCP 服务连接，可将各类工具（如文件系统、网络抓取等）作为 Agent 工具使用。 Python 包（pip install mcp-agent），配置简单，可用 YAML 文件指定 MCP 服务器。运行时无需额外部署，只需 Python 环境。 通过“Augmented LLM”抽象接入任意 LLM，默认支持 OpenAI（可指定 GPT-4o）。用户可替换为其它模型提供者或自定义 LLM 客户端。 专注多代理协作：内置多种工作流模式（并行、路由、意图分类、OpenAI Swarm 等），完美复刻 Anthropic 提出的模式。可轻松实现并行评分、代理级联等复杂任务。 框架本身即插件化，可通过添加新的 MCP 服务器或自定义 Agent 逻辑扩展功能。所依赖的扩展主要通过配置不同的 MCP 服务来实现。 目前社区规模较小（Lastmile/Anthropic 团队开发），但开源活跃。文档包括 GitHub 示例和快速上手教程，示例覆盖多种使用场景。 设计轻量，仅依赖 Python 异步和 MCP 通信，相对高效。资源占用与所连接服务数量和并行代理数成线性关系，适合中小规模部署。 适合构建符合“Building Effective Agents”理念的生产系统，尤其擅长多 Agent 编排和复杂工作流。适用于需要稳定、高可控型 Agent 系统的企业应用。
fast-agent 完全原生的 MCP 客户端框架，从零开始面向 MCP 设计。可直接配置并调用任意 MCP 服务（示例中使用内置的“fetch”服务）。 Python 包（通过 uvx pip install fast-agent-mcp 安装），提供命令行工具（fast-agent 命令）快速初始化项目。依赖 UV 管理器，但安装部署与普通 Python 应用类似。 支持 Anthropic（Haiku/Opus）、OpenAI（gpt-4o/gpt-4.1 系列）以及本地模型（如 Ollama 模型）。可在运行时通过参数灵活切换模型，内置多模态（文本、图像、PDF）支持。 内置链式和多 Agent 模式：框架实现了《Building Effective Agents》中的流水线模式，允许定义多个 Agent 串联执行。支持对话式调试和人机交互，便于协作式工作流设计。 强调低代码和可配置：Agents 和 Workflows 通过 YAML/JSON 描述，框架负责解析执行。提供多模态资源和采样功能，易于插入自定义组件。 新兴框架，GitHub 社区快速增长（1000+ 星标），文档活跃（fast-agent.ai 提供教程和参考）。中文资料较少，但英文文档详实。PulseMCP 等媒体有相关介绍。 类似于 mcp-agent 的设计，性能主要取决于所调用的 LLM 和工具服务。多 Agent 并行时会占用更多资源，但对于原型开发来说开销可接受。 适用于快速原型开发和多 Agent 编排场景，非常适合演示和实验性质的应用。低代码特性使其易于快速构建复杂工作流，适合研究人员和开发者快速试验 MCP 工具集成。

Semantic Kernel 简述¶

Semantic Kernel（SK）由微软推出，原生支持 MCP 协议。新版本的 SK（Python 1.28+ 和 .NET）内置了 MCP 客户端功能，可以通过标准配置调用任意 MCP 服务。例如示例中使用 C# 代码通过 npx @playwright/mcp@latest 启动 Playwright MCP 服务，并自动将其提供的工具函数映射为 Semantic Kernel 的插件函数。安装方面，SK 提供 C# 和 Python SDK，C# 版可通过 dotnet add package Microsoft.SemanticKernel 安装，Python 版通过 pip install semantic-kernel 安装，均需要对应的运行时环境。.NET 版内置对 Azure OpenAI 的支持（示例代码中通过 AddAzureOpenAIChatCompletion 配置）；同时微软提供了本地模型连接器（如 Ollama 连接器），可让 SK 调用本地部署的 LLM。SK 主要聚焦于单个智能体和函数调用流程，尚未提供内置的多代理编排功能；但其插件系统强大，可将任何外部服务包装为 KernelFunctions 加入工作流。文档上由微软官方维护，示例代码丰富且权威，社区规模中等。总体而言，SK 性能良好（尤其是 .NET 版），适合企业级部署：如果项目本身在微软生态（如需要与 Azure 服务集成或使用 C#），Semantic Kernel 是稳定且功能齐全的选择。

LangChain 简述¶

LangChain 是一个开源的 Python 框架，本身不直接内置 MCP 支持，但通过官方的 langchain-mcp-adapters 库可以接入 MCP 服务。安装方式简单（使用 pip），但依赖项较多，需要各种 LLM 客户端和工具库。LangChain 对 LLM 的集成非常全面：可直接调用 OpenAI、Anthropic 等云端模型，也可利用 Hugging Face Pipeline 运行本地模型。在多代理和链式工作流方面，LangChain（特别是新推出的 LangGraph）提供了丰富的体系结构支持，如网络拓扑、监督者、层级等多种多 Agent 协作模式。扩展性是其一大优势：拥有超过 600 个官方和社区集成，包括各种外部 API、数据库、工具等，可自定义 Chain、Tool、Memory 等组件。LangChain 拥有最大规模的社区和文档（3000+ 贡献者、广泛教程等），资料多、支持好。性能方面，由于完全基于 Python 实现，运行效率依赖于用户代码优化；但 LangChain 提供了 LangSmith 等工具用于性能观察和调优。总体而言，LangChain 易于快速迭代并支持生产环境（已有大量企业案例），特别适合需要灵活构建复杂链式推理或多模型组合的应用场景。

AutoGen 简述¶

AutoGen（微软开源框架）主要面向多智能体和代理协作。安装时使用 Python（要求 >=3.10），核心库 autogen-core 可通过 pip 安装；若使用 OpenAI，需要安装相应扩展（如 autogen-ext[openai]）。AutoGen 支持接入主流云端 LLM，也引入了对本地推理引擎（例如 LiteLLM 或 Ollama） 的支持。在多代理设计上，AutoGen 内置了多种应用模式，如 GroupChat 可以组合多个 Agent 协同完成任务，并支持使用 Azure 的 Assistant API 来管理状态和工具调用。框架提供 Agent、技能、工作流等构建块，用户可扩展各种“技能”或工具（如代码执行、检索工具等）。文档方面有官方指南和示例，社区以教育和研究用户为主。性能上，AutoGen 与普通 Python 相似，资源消耗取决于模型和运行时环境。总的来说，AutoGen 适用于那些需要多智能体协作或结合人机交互的场景（如教育、可视化工作流），并且可通过 AutoGen Studio 提供图形化配置界面，降低开发门槛。

mcp-agent 简述¶

mcp-agent 是 lastmile-ai 推出的轻量级 Agent 框架，原生支持 MCP 协议。它自动管理多个 MCP 服务器的生命周期，用户只需在配置文件中指定需要的服务器（如 fetch、filesystem 等），框架会将这些服务器的工具自动纳入 Agent 可调用范围。安装部署非常简单，只需 Python 环境和 pip install mcp-agent。mcp-agent 采用“Augmented LLM”抽象，可附加任何 LLM；示例默认使用 OpenAI（GPT-4o）。该框架内置了丰富的工作流模式（并行、路由、意图分类、OpenAI Swarm 等），对应 Anthropic “Building Effective Agents” 一书中的模式。通过这些模式，开发者可以轻松构建多 Agent 协作的系统。扩展性方面，mcp-agent 本身关注底层逻辑，用户扩展时通常添加新的 MCP 服务或自定义 Agent 行为即可。社区还处于发展初期（主要由 Anthropic/Lastmile 团队维护），但已有丰富示例说明如何构建协作式工作流。总体来看，mcp-agent 框架轻巧高效，非常适合用于基于 MCP 协议的多智能体应用开发，特别适合需要高度可控和可扩展的生产环境。

fast-agent 简述¶

fast-agent 由 evalstate 开发，是首个从头打造的 MCP 原生智能体框架。它以快速开发为目标，提供命令行工具和低代码配置来定义代理和工作流。安装时通过 pip install fast-agent-mcp（或使用 uv 管理环境），框架提供 fast-agent setup 等命令自动生成示例项目。LLM 支持方面，fast-agent 可使用 Anthropic（Haiku/Sonnet/Opus）和 OpenAI（GPT-4o/4.1 系列）模型，并且能灵活切换本地 Ollama 模型。它还支持多模态输入（文本、图像、PDF）。在工作流编排上，fast-agent 内置了多智能体串联执行的模式：开发者可通过简单的装饰器语法定义多个 Agent，并用 @fast.chain 将它们串联。框架还提供交互式调试前端，方便在开发中检查 Agent 对话。fast-agent 的扩展主要通过配置文件完成，框架负责加载和管理组件。该项目目前新兴但活跃（GitHub 已有上千星标），英文文档详尽，社区正在增长。性能方面与其他 Python 框架类似，关注快速迭代和多 Agent 协作的效率。总的来说，fast-agent 非常适合快速构建原型、实验性项目或演示应用，尤其是在需要组合多个代理、集成多种模型和服务的场景中表现出色。

最终推荐¶

综上所述，五个框架各有侧重：如果快速构建原型或需要大量开源支持，推荐使用 LangChain 或 fast-agent；两者均易上手、社区活跃，其中 fast-agent 对 MCP 原生支持更好，特别适合多代理串联应用。对于企业级部署，若项目偏向微软/.NET 生态或需与 Azure 服务深度集成，则 Semantic Kernel 是稳健之选；LangChain 也已被众多企业采用，适合多模型、多数据源场景。若项目核心强调多智能体协作和复杂工作流编排，则 mcp-agent 和 fast-agent 更具优势，它们提供了完整的多 Agent 模式和工作流范式；LangChain（LangGraph）在这一方面也颇为成熟。最后，如果需要可视化界面或教育/研究用途，AutoGen (及其 Studio) 提供了低代码图形化开发流程。每个框架均可作为 MCP Playwright 客户端使用：其中 SK 和 fast-agent 对 Playwright MCP 服务器的支持更明确（如上文示例），LangChain 需借助适配器，mcp-agent 和 AutoGen 则可通过通用 MCP 调用机制接入。用户可根据自身语言栈、部署需求和工作流复杂度选择最合适的工具。
